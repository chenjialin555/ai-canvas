# 2. 它做什么？

函数体是一个 **`switch (cmd.type)`**：每种 `type` 对应调用 **`useEditorStore.getState()`** 上的某个方法。

例如：

```text
copy      → store.copy()
paste     → store.paste()
removeSelected → store.removeSelected()
toggleLock → 先读当前元素，再 updateElement
```

注释写得很直白：**单一入口分派到 Zustand**，内部仍是现有 slice 与 `commitHistory` 语义。

---

# 3. 为什么不直接在 JSX 里写 `useEditorStore.getState().copy()`？

可以，但项目选择了 **命令对象** 模式，好处是：

```text
1. ContextMenu、快捷键、未来宏 / 遥测，都传同一种 cmd 结构
2. 类型上 ElementCommand 把「允许的操作」收窄，搜引用方便
3. 以后若在「执行前」加统一拦截（日志、权限），只改这一处
```

你作为读者：**把 executeElementCommand 当成「对 store 的批量别名」即可**。

---

# 4. `history` 参数

`updateElement` 分支里可以看到：

```ts
store.updateElement(cmd.id, cmd.patch, {
  history: cmd.history ?? true,
});
```

人话：

```text
有的命令希望默认入撤销栈；
少数场景可以传 history: false（与第六课「手势中间帧不入栈」同一思想）。
```

---

# 5. 和 `App.tsx` 的关系

第十二课里 **`ContextMenu`** 大量回调写的是：

```tsx
executeElementCommand({ type: "copy" });
```

而不是每个 `onXxx` 里复制粘贴一遍 `useEditorStore.getState()` 逻辑 —— **减少重复、统一语义**。
