# 第九课：画布上的 AI 工作流节点（`aiNodes` / `edges`）

这一课只解决一件事：

```text
画布上那些「可连线、可运行」的 AI 块，在前端是怎么串起来的。
```

读完你会知道：**数据在哪、谁画出来、点「运行」后代码往哪走**。  
（不要求你先读完 Python 后端。）

> 系列目录见 [README.md](./README.md)。上一课：[8.md](./8.md)（`/api` 代理）。下一课：[10.md](./10.md)（工程 JSON）。

---

# 1. 数据放在哪一页、哪两个数组？

每一页 `Page` 上，和普通图层并列的有（名称以 `src/editor/types.ts` 为准）：

```text
elements   — 普通画布元素（图、字、矩形…）
aiNodes    — AI 工作流节点实例（一块一块的「节点」）
edges      — 节点之间、或节点与图片端口之间的有向连线
```

**真相源仍在 Zustand**：`useEditorStore` 里的 `pages` → 当前 `activePageId` 对应那一页的 `aiNodes` / `edges`。

人话：

```text
拖动节点、连线、跑节点，最后都是改 store 里这一页的数组。
```

---

# 2. 「节点类型」从哪来？

类型定义在 **`src/workflow/`**（如 `WorkflowNodeDefinition`：端口、`executor` 等）。  
**注册表**在：

```text
src/workflow/nodeRegistry.ts
```

这里维护了当前项目里 **有哪些节点 type**（如 inpaint、upscale、`output-view` 等）。  
新增一种节点时，通常要在这里注册，并在 `src/workflow/nodes/` 里写定义。

---

# 3. 谁在屏幕上画节点？

**Konva 视图**主要在：

```text
src/components/workflow/WorkflowNodeView.tsx
```

它读取 **某个 `WorkflowNode` 实例**（位置、标题、状态、预览图等），负责端口、拖拽写回位置、运行按钮等 UI。

人话：

```text
Store 里只有数据；WorkflowNodeView 负责「长得像节点」的那块 canvas UI。
```

# 4. 连线时你在跟哪几个 store 字段打交道？

架构文档里概括过（你读 `workflowSlice` 也能看到名字）：

```text
workflowConnecting   — 从端口拖线时的临时状态（起点、指针位置、数据类型）
workflowNodePicker   — 松手后弹出「创建哪种节点」的选择器
```

相关方法大意包括：

```text
startWorkflowConnecting / updateWorkflowConnectingPointer / cancelWorkflowConnecting
createWorkflowNodeFromPicker
addWorkflowEdge / removeWorkflowEdge
```

不要求背名字：**DevTools / 全局搜索**能搜到调用处即可。

# 5. 点击「运行」：`runWorkflowNode` 的三条路

核心逻辑在 **`src/editor/store/slices/workflowSlice.ts`** 的 `runWorkflowNode`。

你只要记住 **分支结构**：

## 5.1 纯前端同步：`output-view`

若节点 `type === "output-view"`：

```text
不请求后端。
从上游连线解析出图像 URL，直接写回本节点的 outputs，并 commitHistory。
```

适合「只把上游结果展示出来」的节点。

## 5.2 不需要跑：`executor === "none"`

若节点定义里 `executor === "none"`（且不是上面那种特殊处理），**直接 return**。

## 5.3 走后端：`POST /api/workflow/run-node`

否则大致顺序是：

```text
1. serializeWorkflowInputsForApi — 把当前页 + 连线收成后端要的 inputs
   （若失败 → 节点 status error，结束）
2. updateWorkflowNode(..., { status: "running" })
3. executeWorkflowNodeRemoteRun — 内部调用 postWorkflowRunNode
4. 成功 → 把 outputs 写回节点、commitHistory
   失败 → status error + 错误信息
```

HTTP 封装在：

```text
src/ai/workflow/api/workflowApi.ts  →  postWorkflowRunNode("/api/workflow/run-node", ...)
```

这与 **第八课** 的「浏览器只打 `/api/...`」一致。

# 6. 跑出来的图怎么落到画布上？

Store 里有 **`sendWorkflowResultToCanvas`**（名以你仓库为准）：把节点某路 **image 输出** 转成新的 **`ImageElement`** 加到当前页。

人话：

```text
工作流节点 = 中间站；「落到画布」= 再调一次 addElement 那类逻辑。
```

---

# 7. 和「单步生图弹窗」的区别（防混）

| 场景 | 典型入口 | 常见 API |
|------|----------|----------|
| 侧栏/弹窗 **单步**生图 | `AiGenerateModal` 等 | `POST /api/generate-image` |
| 画布上 **节点**执行 | `WorkflowNodeView` 点运行 → `runWorkflowNode` | `POST /api/workflow/run-node` |

第十一课会专门走一遍单步链。

# 10. 本课总结

```text
aiNodes + edges     = 工作流图在每一页上的存法
WorkflowNodeView    = Konva 上画节点
runWorkflowNode     = 运行入口：同步展示 / 不跑 / 远端跑
postWorkflowRunNode = 远端跑时真正发 HTTP
sendWorkflowResultToCanvas = 把节点输出变成画布图片元素
```
