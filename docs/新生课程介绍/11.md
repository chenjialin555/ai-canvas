# 第十一课：单步 AI 生图（弹窗 → `/api/generate-image`）

这一课与 **第九课** 区分：不讲画布节点执行，只讲 **「文生图 / 图生图 / 局部重绘」弹窗那条链**。

```text
填表单 → 组装 payload → POST /api/generate-image → 拿 URL → 落到画布或替换图层
```

# 2. 真正发请求的地方：分层读

建议按这个顺序搜：

```text
1. generationApi.ts   — postGenerateImage → POST /api/generate-image
2. generationService.ts — 业务：组 payload、处理返回、与 store/蒙版导出等协作
3. AiGenerateModal.tsx — 用户点了「生成」后调谁
```

**第八课** 说过：URL 写成 **`/api/generate-image`**，开发时由 Vite 代理到 FastAPI。

---

# 3. 和蒙版、裁剪的关系（建立直觉）

架构文档概括过：

- **局部重绘（inpaint）** 往往需要 **蒙版栅格化**（笔划 → `canvas` → dataURL → 再随请求走 OSS/网关）。
- **裁剪**影响的是 **`ImageElement` 上 crop 相关字段**与导出逻辑；请求前用的是「当前编辑器理解的图」。

你作为前端新手：**先知道「弹窗确认前会调一堆准备函数」**，不必第一遍就读完 `maskRasterize.ts`。

---

# 4. 后端返回后前端做什么？

常见模式：

```text
HTTP 200 且 body 里带有图像 URL（或项目约定的字段）
→ 调 store：新增 ImageElement 或 replaceImageKeepFrame 等
→ 弹窗关闭或保留预览
```

具体字段名以 **`generationService.ts`** 与后端契约为准。

---

# 5. 与工作流节点的 API 对比（背这张表就够）

| 能力 | HTTP |
|------|------|
| 单步生图弹窗 | `POST /api/generate-image` |
| 画布节点执行 | `POST /api/workflow/run-node` |

两条线都可能经过 **OSS / 模型网关**；差别主要在 **payload 形状** 和 **谁触发**（弹窗 vs `runWorkflowNode`）。

# 8. 本课总结

```text
AiGenerateModal = 单步 UI 入口之一

```
