# 第六课：Zustand，全局状态

> 系列目录见 [README.md](./README.md)。上一课：[5.md](./5.md)（TypeScript）。下一课：[7.md](./7.md)（Konva）。

这一课我们只学一个东西：

```text
Zustand
```

你的项目里，**画布上有什么图、选中谁、缩放平移、撤销重做、工作流节点**，几乎都在 **一个全局 Store** 里。  
看不懂 `useEditorStore`，就看不懂大半业务代码。

---

# 1. 为什么要有「全局状态」？

`useState` 只管**当前组件自己**的数据。

但你的编辑器是：

```text
顶栏要读「当前工具」
侧栏要读「选中元素的属性」
画布要读「所有元素 + 缩放平移」
浮动条要读「选中框位置」
```

如果全用 props 一层层往下传，会写成「面条代码」。

所以项目用 **Zustand**：一个**全局**的 JavaScript 对象 + 一套**订阅 / 更新**机制。  
你可以先把它想成：**一份全应用共享的数据 + 更新函数**，多个组件都能读、都能触发刷新；这和你在原生里自己维护一个全局对象再手写「通知界面更新」是同一类问题，只是 Zustand 帮你做好了。

---

# 2. 一句话：Zustand 是什么？

```text
Zustand = 用 create() 造一个全局 store，
         用 useEditorStore(selector) 在 React 里订阅其中一部分，
         用 getState() 在非 React 代码里读/写。
```

---

# 3. 你的项目里 store 叫什么？

入口在：

```text
src/editor/store/index.ts   ← 真正 create 的地方
src/editor/store.ts         ← 对外 re-export，很多地方 import 这里
```

对外使用的名字是：

```tsx
import { useEditorStore } from "../editor/store";
```

（路径按文件所在目录调整。）

---

# 4. store 是怎么「造」出来的？

在 `src/editor/store/index.ts` 里，核心形状是：

```tsx
import { create } from "zustand";

export const useEditorStore = create<Store>()((set, get) => ({
  ...getInitial(),
  // 初始字段 + 各 slice 合并进来的方法
  ...createHistorySlice(set, get),
  ...createElementSlice(set, get),
  // ... 其它 slice
}));
```

人话：

```text
create<Store>() 创建一个「类型为 Store 的全局状态机」。
第二个参数是 (set, get) => { ... }，返回「初始 state + 所有方法」。
```

- **`set`**：更新状态（把旧 state 合并成新 state；底层细节可先不管）。
- **`get`**：在方法内部读取**当前最新**的整包 state（避免闭包拿到旧值）。

项目把不同职责拆成多个 **slice 文件**（`slices/*.ts`），最后 **展开合并** 进同一个 `useEditorStore`。  
你不需要一次读完所有 slice，只要知道：**所有方法最后都挂在同一个 store 上**。

# 5. 在 React 组件里怎么用？

最常见两种：

## 5.1 订阅一部分数据（推荐）

```tsx
const selectedIds = useEditorStore((s) => s.selectedIds);
const zoom = useEditorStore((s) => s.zoom);
```

人话：

```text
只要 selectedIds 或 zoom 变了，这个组件会重新渲染。
其它无关字段变了，不会拖累这个组件。
```

括号里的 `(s) => s.xxx` 叫 **selector（选择器）**。  
**尽量写细**：只选你真正要显示在界面上的字段。

## 5.2 订阅整个 store（少用）

```tsx
const state = useEditorStore();
```

人话：

```text
store 里任何字段一变，这个组件都可能重渲染。
```

新手调试用可以，正式组件里容易变慢。

---

# 6. `getState()` 是什么？为什么事件里常见？

有时你在 **回调、定时器、原生事件、Konva 事件** 里要调用 store 方法，这时**没有 React hook 上下文**，就写：

```tsx
useEditorStore.getState().setFloatingToolbarSuppressed(true);
```

你项目里 `StageCanvas.tsx` 就有类似用法。

人话：

```text
getState() = 直接拿到当前 store 快照 + 所有方法，
不触发「订阅重渲染」的那套 hook 逻辑。
```

适合：**「我只想改状态，这个组件未必订阅了那个字段」** 的场景。

# 7. 和「自己写一个全局 store」对照（心智模型）

| 自己用原生 JS 维护 | 本项目 (Zustand) |
|---------------------|------------------|
| 模块级 `let state = {...}` | `create()` 内部的 store state |
| 手写 `subscribe(listener)` | `useEditorStore(selector)` 订阅 |
| 手写 `setState(partial)` | `set(...)` / 各 slice 里调用 `set` |
| 在事件回调里直接读 `state` | `useEditorStore.getState()` |

# 8. 你的 Store 里大概有什么？（按「看代码」优先级）

结合 `doc/PROJECT_ARCHITECTURE.md`，你先记这几类即可：

## 8.1 持久化相关的「工程数据」

```text
pages          — 多页画布，每页有 elements / aiNodes / edges
activePageId   — 当前正在编辑哪一页
zoom / pan     — 视口缩放与平移
```

这些数据会参与 **localStorage 自动保存**（防抖写入，键名见下一节）。

## 8.2 交互相关的「选区与工具」

```text
selectedIds                 — 当前选中的画布元素 id 列表
selectedWorkflowNodeIds     — 当前选中的 AI 工作流节点
tool / editingTextId 等     — 视口 slice 里
```

## 8.3 大量「方法」

例如（名字以你仓库为准）：

```text
updateElement      — 改某个元素的字段（位置、文字内容…）
addElement         — 新增元素
commitHistory / undo / redo
runWorkflowNode    — 跑某个 AI 节点
exportProjectJSON / loadProjectJSON
```

**原则**：能在 store 里改的，尽量在 store 里改；组件只负责 **展示 + 把用户操作翻译成 store 调用**。

# 9. localStorage 自动保存（你只要知道现象）

常量里有：

```text
STORAGE_KEY = "AI_CANVAS_PRO_PROJECT_V2"
```

机制大意是：

```text
每次 setState 后，调度一次「防抖写入」（约 600ms 合并多次变更），
把工程 JSON 写进 localStorage。
```

人话：

```text
你改画布 → store 更新 → 过一段时间自动落盘到浏览器本地。
```

如果存不下（配额爆了），项目里会对特别大的 `data:` 图片做瘦身占位，**避免整页卡死**；细节以后读 `persistence/localStoragePersist.ts` 即可。

# 10. Immer 的 `produce` 是干什么的？

在 `elementSlice.ts` 里你会看到：

```tsx
set(
  produce<Store>((state) => {
    const page = state.pages.find((p) => p.id === state.activePageId);
    const el = page?.elements.find((item) => item.id === id);
    if (!el) return;
    Object.assign(el, patch);
  }),
);
```

人话：

```text
produce 让你在回调里「像直接改对象一样写代码」，
实际上 Zustand 收到的是「不可变更新」后的新树。
```

对新手：先把它当成 **「深层对象的安全赋值工具」**，不必先学 Immer 全部 API。

# 11. `updateElement` 为什么第二个参数是 `Partial<CanvasElement>`？

第五课学过 `Partial<T>`：**可以只传要改的字段**。

```tsx
updateElement(id, { x: 100, y: 200 });
```

不需要把整个元素对象手写一遍。

# 12. 历史记录（撤销 / 重做）和 `updateElement` 的关系

很多会改画布的方法里，会先 `commitHistory()`，再 `set(produce(...))`。

人话：

```text
改之前先拍一张快照，方便用户按 Ctrl+Z 回去。
```

有些高频中间状态（例如拖动的每一帧）会 **关掉 history**，只在手势开始/结束各记一次；这是性能优化，你在 `interactionGestureHistory.ts` 里能看到痕迹。

# 17. 本课总结

你只要记住：

```text
useEditorStore = 整个编辑器的大脑（数据 + 方法）
selector       = 只订阅自己关心的字段，少重渲染
getState()     = 在事件 / 非 React 逻辑里读写 store
slice + Immer  = 大项目把 store 拆文件 + 安全改深层数据
```
