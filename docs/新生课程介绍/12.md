这一课只回答：

```text
打开 src/app/App.tsx，这一大坨 JSX 到底在干什么？

```

> 系列目录见 [README.md](./README.md)。上一课：[11.md](./11.md)。下一课：[13.md](./13.md)。

# 1. 文件顶部的注释已经剧透了

`App.tsx` 开头写得很清楚（大意）：

```text
布局骨架       → AppShell
全局编辑数据   → useEditorStore（页面、元素、工作流…）
纯 UI 状态     → useAppModals（弹窗开关、右键菜单、侧栏 Tab…，不写进 Zustand）
导入导出 / 导出画布 → 各自 hook
```

**背诵版**：**可序列化、要撤销、要多组件共享的「工程真相」→ store；**  
**只控制「此刻弹不弹窗、菜单位置」→ modals hook。**

---

# 2. 三个「横向能力」hook

在 `App()` 里通常先看到：

```tsx
const modals = useAppModals();
const project = useProjectImportExport();
const { exportStage } = useStageExport(stageRef);
```

| Hook | 人话 |
|------|------|
| `useAppModals` | 素材库 / AI 弹窗 / 统一图片编辑器 Tab / 右键菜单 state / 侧栏 tab … |
| `useProjectImportExport` | 隐藏 `<input type=file>` + 导入导出 `ProjectJSON` |
| `useStageExport` | 需要 `stageRef` 时，把整页 Stage 导出成图 |

它们和 **store 并列**：App 负责 **接线**，不把业务全写进一个 5000 行函数里。

---

# 3. `AppShell`：五个槽位

```tsx
<AppShell
  onRootMouseDown={...}
  topBar={...}
  leftSidebar={...}
  canvasArea={...}
  rightSidebar={...}
  overlays={...}
/>
```

第二课学过：**slot 都是 props 传进去的 ReactNode**。  
这里只是 **真实项目版**：`topBar` 里是 `<TopBar ... />`，`canvasArea` 里是 `<CanvasArea ... />`。

`onRootMouseDown`：点击应用根区域时 **顺手关掉右键菜单**（若正在显示）。

---

# 4. `CanvasArea`：画布区回调的「总线」

你会看到一长串 `onOpenCropEditor`、`onContextMenu`、`onFloatingMask` 等。

典型模式：

```text
子组件（StageCanvas / FloatingToolbar）只负责说「用户想打开裁剪」
App 决定：调 modals.openImageEditor(id, "crop")
```

例如（与源码一致的思想）：

```tsx
onOpenCropEditor={(id) => modals.openImageEditor(id, "crop")}
onFloatingMask={(id) => modals.openImageEditor(id, "mask")}
```

**单步 AI** 从浮动条进来时，可能先 `setSelectedIds`，再 `setAiOutputMode` + `setAiOpen(true)` —— 都在 `App.tsx` 里 **集中写好**，子组件不用知道 modals 有几个字段。

# 5. `useEditorStore` 在 App 里读什么？

常见写法：

```tsx
const selectedIds = useEditorStore((s) => s.selectedIds);
const page = useEditorStore((s) =>
  s.pages.find((p) => p.id === s.activePageId),
);
const selected = useMemo(
  () => page?.elements.find((el) => el.id === selectedIds[0]),
  [page?.elements, selectedIds],
);
```

人话：

```text
App 需要「当前选中的是不是图」来决定浮动条「连接 AI」的提示位置，
所以在这里订阅一小片 store，并用 useMemo 派生出 selected。
```

**不要在 App 里订阅整棵 store**（`useEditorStore()` 无 selector），否则任何小改动都可能让整个 App 重渲染。

---

# 6. `overlays`：右键菜单 + 所有弹窗 + Toast

结构大致是：

```tsx
overlays={
  <>
    <ContextMenu ... />
    <EditorModals modals={modals} project={project} />
    {modals.toolbarToast && <div className="toolbar-toast">...</div>}
  </>
}
```

- **`ContextMenu`**：吃 `modals.contextMenu`，里面各种 `onCopy` / `onDelete` … 很多走 **`executeElementCommand`**（第十三课）。
- **`EditorModals`**：素材库、设置、`AiGenerateModal`、`ImageEditorModal`、隐藏 file input 等 **集中挂载**。
- **Toast**：用 `toolbarToastTimerRef` 做自动消失（第四课 `useRef`）。

---

# 7. `EditorModals` 做什么？

它接收 **`modals` + `project`** 两个对象，避免在 `App.tsx` 里再堆几十行 modal JSX。

里面你能看到：

```text
隐藏的 JSON file input → project.importJSON
隐藏的换图 file input → replaceImageFitFrame + store
LibraryPanel / AiGenerateModal / ImageEditorModal / 各种 Settings
```

**`ImageEditorModal`** 的 `open` 由 `!!modals.imageEditor` 决定；`imageId` / `tool` 从 `imageEditor` 状态里取 —— 这就是第三课「布尔 + 对象联合控制弹窗」的实战版。

---

# 8. `useAppModals` 里有什么？（不必全背）

打开 **`src/app/hooks/useAppModals.ts`**，你会看到大量 **`useState`**：

```text
libraryOpen、aiOpen、imageEditor、contextMenu、rightTab …
```

返回对象里还有 **`openImageEditor` / `closeImageEditor`**、`addToAiChat` 等封装。

对照第三课：

```text
第三课：一个 Modal 一个 open 布尔值
本仓库：很多弹窗 → 抽成一个 hook，App 只解构 modals.*
```

---

# 9. 为什么「连接 AI 节点」要动 `useEditorStore.getState()`？

例如 `openWorkflowPickerWithToast` 里：

```tsx
useEditorStore.getState().openWorkflowNodePickerAtWorld(worldX, worldY);
```

人话：

```text
打开节点选择器是「编辑器全局行为」，数据在 workflow slice；
不必让 FloatingToolbar 直接 import store，也可以由 App 统一调；
用 getState() 和第六课一致。
```

（具体是否 refactor 到子组件是个人口味；**当前结构是「App 当总线」**。）

# 12. 本课总结

```text
App.tsx     = 布局槽位 + 回调总线 + 少量派生状态（selected）
useEditorStore = 工程与画布真相源
useAppModals   = 弹窗 / 菜单 / 侧栏 UI 状态
EditorModals   = 所有「浮在画布上」的对话框集中出口
```
