# 第十九课：`initPlatform` 与 API 根地址 — Web、桌面与 `fetch`

这一课把 **`main.tsx` 里的 `initPlatform()`** 和 **第八课** 的 `/api` 代理串起来：你写 `fetch('/api/...')` 时，**完整 URL 到底是谁拼的**。

> 系列目录见 [README.md](./README.md)。上一课：[18.md](./18.md)。下一课：[20.md](./20.md)。

---

# 1. 统一入口：`setApiBaseUrl` + `resolveApiFetchUrl`

文件：**`src/ai/api/client.ts`**（大意）：

```text
模块级变量 apiBaseUrl，初始为 ""
setApiBaseUrl(url) 会去掉末尾斜杠后写入
apiUrl(path)：
  - 若 apiBaseUrl 为空 → 直接返回以 / 开头的 path（例如 "/api/generate-image"）
  - 若非空 → 拼成 `${apiBaseUrl}${path}`（绝对根 + 相对路径）
resolveApiFetchUrl：已是 http(s) 则原样；否则走 apiUrl
```

`apiPostJson` 等（`aiClient.ts`）最终都通过 **`resolveApiFetchUrl`** 得到真正 `fetch` 的字符串。

---

# 2. Web 开发：为什么 `getApiBaseUrl()` 经常是空？

**`src/platform/web.ts`** 里：

```text
读 import.meta.env.VITE_API_BASE_URL
若没配或为空 → 返回 ""
```

于是 **`apiBaseUrl` 保持 ""**，请求路径保持 **`/api/...` 相对当前站点** —— 在 **`npm run dev`** 时由 **Vite 代理**转到本机后端（第八课）。

人话：

```text
开发时多数情况「不配环境变量」也能跑，因为走的是同源相对路径 + 代理。
```

---

# 3. 何时需要配 `VITE_API_BASE_URL`？

典型场景：

```text
前端与后端不在同一站点（无代理）
Electron / 打包后打开 file:// 或固定端口组合
生产环境静态资源与 API 域名分离
```

此时 `getApiBaseUrl()` 返回 **`https://api.example.com`** 一类，`fetch` 会变成 **绝对 URL**。

---

# 4. 桌面端：`detectPlatform`

**`src/platform/index.ts`**：

```ts
if (typeof window !== "undefined" && window.desktopAPI) {
  return "desktop";
}
```

有 **`window.desktopAPI`** 就走 **桌面适配器**（`createDesktopPlatformAdapter`），否则 Web。  
桌面适配器里 `getApiBaseUrl` 的实现通常与 **预加载脚本注入的基地址** 有关（细节以 `src/platform/desktop.ts` 为准）。

---

# 5. 和 `vite.config.ts` 代理的关系（防混）

| 机制 | 作用 |
|------|------|
| **`Vite proxy /api`** | 开发服务器把浏览器里的 **`/api`** 转到后端端口 |
| **`VITE_API_BASE_URL` + setApiBaseUrl** | 告诉 **`fetch` 拼出来的字符串**要不要加「另一套域名前缀」 |

两者 **不矛盾**：多数本地开发 **`apiBaseUrl` 为空**，完全依赖 **相对路径 + 代理**。

# 8. 本课总结

```text
initPlatform   → setApiBaseUrl（来自适配器）
apiBaseUrl 为空 → 相对 /api，交给 Vite 代理或同源部署
apiBaseUrl 非空 → 绝对地址直连另一主机
```
